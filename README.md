Part 2 - Docker & Containerization

Difference between Dockerfile, Docker Image and Docker Container

Dockerfile:

A Dockerfile is a text file that contains instructions to build an application image. It tells Docker what steps to follow.

Docker Image:

A Docker image is a package that contains the application code, libraries and all required dependencies.

Docker Container:

A Docker container is a running instance of a Docker image. It is where the application actually runs.

How would you reduce the image size?

If my Docker image is too large, I would:

* Use a lightweight base image such as python:3.11-slim.
* Remove unnecessary packages and files.
* Use a .dockerignore file.
* Use multi-stage builds when needed.

Part 3 - Kubernetes (EKS Basics)

Difference between Pod, Deployment and Service

Pod:

A Pod is the smallest unit in Kubernetes that runs one or more containers.

Deployment:

A Deployment manages Pods and makes sure the required number of Pods are always running.

Service:

A Service provides a stable way for users to access the application running inside Pods.

Why do we use EKS instead of running Kubernetes on VMs?

Amazon EKS makes Kubernetes easier to manage.

With EKS:

* AWS manages the Kubernetes control plane.
* It is easier to scale applications.
* It provides high availability.
* It reduces maintenance work.

Running Kubernetes directly on VMs requires more manual management.

Part 4 - CI/CD Pipeline

How would the pipeline change if we deploy to Kubernetes?

The pipeline would work like this:

1. Developer pushes code to GitHub.
2. GitHub Actions builds the Docker image.
3. The image is pushed to DockerHub.
4. Kubernetes deployment files are updated.
5. The application is automatically deployed to EKS.

Part 5 - Monitoring and Logs

Difference between Metrics, Logs and Traces

Metrics:

Metrics are numerical values that show system performance such as CPU usage, memory usage and request count.

Logs:

Logs are records of events and errors generated by an application.

Traces:

Traces show the complete path of a request as it travels through different services.

How would you debug a crashed Kubernetes Pod?

I would follow these steps:

1. Check whether the Pod is running.
2. View the Pod details.
3. Check the application logs.
4. Check Kubernetes events.
5. Enter the Pod and inspect the issue if needed.

Commands:

kubectl get pods

kubectl describe pod 

kubectl logs 

kubectl get events

kubectl exec -it  – /bin/sh

Which monitoring tools would you suggest for AWS EKS?

* Prometheus for collecting metrics.
* Grafana for creating dashboards.
* CloudWatch for AWS monitoring and logs.
* ELK Stack for log analysis.

Part 6 - Problem Solving Scenario

High-Level Approach

If I need to set up a new microservice in AWS EKS, I would follow these steps:

1. Get the application code from GitHub.
2. Create a Dockerfile and containerize the application.
3. Build and test the Docker image.
4. Push the image to DockerHub.
5. Create Kubernetes deployment and service files.
6. Create a CI/CD pipeline using GitHub Actions.
7. Configure automatic deployment after merging into the main branch.
8. Set up monitoring and logging for the development team.
